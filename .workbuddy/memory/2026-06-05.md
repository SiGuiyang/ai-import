### 将规则数据存储迁移至 Neon 数据库

**问题**：`src/lib/rule-store.ts` 使用 `localStorage` 存储规则，导致规则数据仅保存在浏览器端，无法跨设备/服务端共享，且后端 Neon 数据库 `parse_rules` 表未被使用。

**改造内容**：
- `src/lib/rule-store.ts`：全部函数改为 `async`，操作替换为 `fetch /api/rules` API 调用（GET/POST/PUT/DELETE）
- `src/app/rules/page.tsx`：所有调用改为 `await`，加载/删除/复制规则改为异步，Table 加入 `loading` 状态
- `src/app/page.tsx`：`refreshRules`、`handleSaveAiRule` 改为 async/await，文件选择后自动触发规则加载
- `src/components/RuleEditor.tsx`：`handleSave` 改为 async/await

**数据流向**：前端 → `/api/rules` → Neon PostgreSQL `parse_rules` 表（存储完整 `rule_json` JSONB）

**注意**：使用前需在环境变量中配置 `DATABASE_URL`（Neon 连接串）。

### 支持所有 demos 文件格式并正确解析

**背景**：`/Desktop/demos` 目录中有 5 种 Excel 文件 + 1 个 PDF：

- `src/app/api/ai/generate-rule/route.ts`：PDF 分支改用真实文本，AI 可理解 PDF 内容生成规则
- `src/lib/rule-engine/index.ts`：PDF 解析引擎重写，按序号识别表格行，先提取头部 key-value 字段，按列数自动分配字段

**TypeScript 修复**：

- `raw-data.ts`：`extractExcelData` 接受 `ArrayBuffer | Buffer`
- `rule-engine/index.ts`：修复 `_sheetName` 注入导致的 ParsedRecord 类型错误，修复 undefined key 的 includes 调用
